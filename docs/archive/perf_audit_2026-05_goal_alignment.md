# Performance Audit — Goal Alignment Verdict

**Branch:** `stt-quality` · **Evidence:** 20 recorded runs (`logs/runs/`), RTX 4090 laptop, gemma3:4b/Ollama · all numbers re-aggregated from the run bundles and reproduce exactly. Goal anchor (user's words): *a low-latency assistant that supports **interrupt** + **add-on** to a specific question/task, with a **smart decision layer** that distributes work so it **never gets stuck** and never spawns too many processes.*

---

## 1. Verdict

**Is it performant enough? Yes for the thing you actually shipped — a fast, single-tier, warm voice loop — and the engineering underneath is genuinely good. But it is *not yet* the assistant you described.** Three of your four goals are only half-built or are running in a degenerate configuration that your own metrics can't see. The warm hot path is fast (`first_audio` p50 **0.195s**, p95 **0.835s**). The interrupt *cut* machinery is correct. The concurrency core is well-engineered and never leaks. The gaps are at the edges you feel most: (1) the single largest fixed latency cost — the **0.8s endpoint wait** — is **structurally invisible** to every metric, so "p50 0.20s" under-reports real perceived latency by ~0.8s; (2) barge-in *detection* has never once cancelled real work in the field — it only ever self-interrupted on TTS echo; (3) ADD-ON/continuation is **unimplemented**; (4) the "smart decision layer that distributes work" collapses to a **single fast tier** in the shipped config (0/47 LLM calls escalated). None of these is a deep architectural flaw — the substrate to fix them all is already present.

| Dimension | Verdict | One-line why | Headline number |
|---|---|---|---|
| **A. Low latency** | **adequate** | Warm hot path is genuinely fast, but the dominant fixed cost (endpoint) is unmeasured and turn-1 is cold | `first_audio` warm p50 **0.195s** / p95 **0.835s**; `endpoint_latency` **0.000s** across all 113 turns (invisible) |
| **B. Interrupt + add-on** | **at-risk** | The *cut* is solid; *detection* never fired on a real interrupt (only echo), and ADD-ON is unimplemented | `barge_in_latency` n=**3**, all ~0.0002s; **0/47** LLM calls cancelled; 22 finals → 21 independent cold tasks (run-232356) |
| **C. Smart decision layer** | **at-risk** | Deterministic control classifier is excellent; semantic escalation never happens in shipped config | **125/125** "answering" decisions on fast tier, **0** main tier; router threshold fix is uncommitted working-tree only |
| **D. Distribution / never-stuck / not-too-many-processes** | **adequate** | Real global task ceiling + leak-free; but the only liveness monitor cries wolf and capacity counts liveness not progress | `MAX_ACTIVE_TASKS=6` enforced; stress 1000-turn +1.2MB RSS no leak; watchdog false-positives "LLM stuck" on answered turns |

## 2. What's genuinely strong (protect these)

- **Sentence-streaming TTS overlaps playback with generation** — `core/capabilities.py:128-153` (`_stream_and_speak`) + `drain_complete_sentences`. **On by default in every shipped profile** (`config.json:235` `tts.streaming=true` lives at top level, survives deep-merge; both `python -m core` and `remote/worker.py` enable it). Keeps `first_token_to_audio` p50 at **0.14s**.
- **Barge-in *cut* is correct and fast** — `core/engines/sherpa.py:494-519` (`stop_speaking`) aborts the PortAudio buffer under `_out_lock` and stamps `BARGE_IN_STOP` **only when it actually cut audio**. Epoch-gated cancel (`supervisor.py:114-163`) bumps `speech_epoch` under `_cancel_lock` and **never holds the lock across blocking work**.
- **Interrupt bypasses the event bus** (load-bearing) — `core/runtime.py:254-256` (`_on_barge_in`) and `:275-277` (`_on_command` stop) call `supervisor.cancel_all()` + `engine.stop_speaking()` **directly on the audio thread before any `bus.publish`**. This is *why* interrupt latency isn't gated by a slow handler. **Document and keep it.**
- **Real global task ceiling** — `DEFAULT_MAX_ACTIVE_TASKS=6` (`tasks.py:59`) consulted by `supervisor._should_queue` via `tasks.at_capacity()` across all modes. Verified leak-free: stress `turns --n 1000` → RSS **+1.2MB**, threads 3→1; `bargein --n 300` → 300/300, 0 hung.
- **Epoch-stamped stale-sentence drop** — `supervisor.tts_request_allowed` (`:140-163`) discards sentences from a cancelled turn even after `TASK_COMPLETED` dequeues. This is the correct defense against the Pipecat #1323/#4466 "bot repeats itself after interrupt" class.
- **Ollama `keep_alive=-1`** on the desktop profile (`config.json:118`) pins residency → warm TTFT collapses to ~0.27s.
- **Conservative speaker gate** — `core/engines/speaker_gate.py:107-116` no longer blindly fails open when unenrolled; gates on output margin. `speaker_gate_input=true` is the shipped default.

## 3. Blockers (confirmed, adversarially verified — hardest-hitting first)

### B1 — Tier router never escalates: the system is effectively fast-only `[dec-1]`
- **What:** The "smart layer that distributes work" does not distribute. Every turn answers on gemma3:4b.
- **Evidence:** `core/routing.py:233-277` base `threshold=0.5` (`config.json:15`, **same in HEAD and working tree**). Reproduced live: `"compare python and rust"`=0.18, `"explain how a transmission works"`=0.36, `"summarize the meeting notes…"`=0.18, `"debug this function"`=0.18 — all FAST. Field: **125/125** "answering" decisions fast tier, **0** main; re-aggregated LLM traffic **100% gemma3:4b, 0 gemma3:12b, 0 cancellations**. Short spoken queries score ~0 because length buckets start at n≥12 words (`routing.py:254-259`) and one complexity marker adds only +0.18 (`:261-262`).
- **Verified nuance:** The working tree adds `_GENERATION_MARKERS` (`routing.py:219`, **+15 lines, uncommitted** — `git show HEAD:core/routing.py` has **0** hits) and `router.threshold=0.3` (only inside the uncommitted `desktop` profile, `config.json:120`). Its `_comment` references **`docs/archive/flow_control_intelligence_2026-05.md` which does not exist.** So escalation works *only* in the uncommitted working tree; the committed code and every recorded run are fast-only.
- **Impact:** Defeats goal C outright in the shipped artifact. Main/research tier, hedging, and ReAct have **zero field activations**.
- **Fix:** Commit `threshold=0.3` to the **base** `llm.router` block (not just a profile), or raise per-marker weight so one reasoning marker clears the bar; add the missing doc; re-record one session showing a real gemma3:12b turn. Long-term: feed RouteLLM (already cited in `routing.py:1-9`) a meaning signal, not just lexical markers.
- **Effort:** **S** (the fix exists, it just needs committing + evidence).

### B2 — ADD-ON unimplemented: a follow-up spawns a competing cold task with no history `[int-3]`
- **What:** Your explicit "ADD-ON to a specific question/task" goal does not exist. A follow-up races a fresh task instead of continuing.
- **Evidence:** `speech_analyzer.decide()` (`speech_analyzer.py:80-137`) is **stateless** — no "continue prior task" branch. `supervisor._execute_decision` (`:243-269`) calls `create_task→_start_task`, never appends/merges. `capabilities.py:257-264`: `assistant.answer` streams **only `query`**; history is injected only via `recall_block`, gated `memory.recall_enabled=false` (default off). Field: run-232356 — **22 finals → 21 independent cold tasks** (`Um`, `But`, `Yes`, `Rob is` each answered cold).
- **Impact:** Defeats the ADD-ON half of goal B. Worse with input_gate off (B-block below) feeding disfluency fragments in.
- **Fix:** (a) Thread `memory.all()[-4:]` into `assistant.answer` **appended after** the constant system prompt (preserve prefix cache); (b) add a continuation path so a new final during an active assistant task **cancels-and-restarts with merged text** (reuse the epoch machinery — discard is free); (c) record only **spoken** sentences to memory on barge-in (`supervisor.py:332` currently writes the full streamed text).
- **Effort:** **L.** OSS: LiveKit turn-detector continuation detection; Pipecat interruption-context-truncation (#2791/#4466).

### B3 — The only never-stuck monitor cries wolf in the exact failure mode that matters `[dist-2]`
- **What:** The watchdog — your single liveness signal, diagnosis-only by design (`watchdog.py:24`) — emits false "LLM stuck" warnings on turns that were answered fine.
- **Evidence:** `metrics.py:130-133` banks an open `TurnRecord` (still missing `llm_first_token`) every time a back-to-back `ASR_FINAL` re-opens a turn → orphan keeps `asr_final` forever with no first-token. `watchdog._check_turns` (`:158-165`) then fires "llm stuck … no llm_first_token after ~11s". The watchdog *does* skip BARGE_IN turns (`:157-159`) — but the metrics orphan carries **no** BARGE_IN stamp, so it still trips. Verified: run-004208 emitted 3 such warnings (turns 0/2/4) while the same run answered 12/12 LLM requests on gemma3:4b.
- **Impact:** Your only never-stuck channel is **least trustworthy in the most common field failure (open-mic echo)** — an operator can't tell a real stall from a benign one. This is a cross-cutting interaction with B-block + dist (see §5).
- **Fix:** Key `TurnRecord`s by task/utterance id instead of re-banking on every `ASR_FINAL`, OR have `_check_turns` skip turns the brain marked intentionally-unanswered (extend the `runtime.py:211-213` INGEST early-return suppression). Then a *genuine* stall becomes distinguishable — optionally add a bounded OTP-style `on_stuck` heal hook mirroring the existing `on_storm` hook (`watchdog.py:215`).
- **Effort:** **M.** OSS: Pipecat `WatchdogQueue`/`WatchdogEvent` (resets timer on legitimate idle waits); Erlang/OTP max-restart-intensity for the bounded heal.

## 5. Cross-cutting gaps / unmeasured risks (what the current metrics CANNOT see)

1. **Endpoint cost is invisible (the whole latency verdict rests on this).** `endpoint_latency`=0.000 across all 113 turns because `SPEECH_END` is stamped == `ASR_FINAL` (`sherpa.py:682-683`). Every "p50 0.20s" number **excludes the ~0.8s the user actually feels**. Also: `tools/specsim`/`bench` budget `FIRST_AUDIO_BUDGET=(1.2,2.5)` *includes* the endpoint the live number throws away — the CI perf gate is ~0.8s over-optimistic `[lat-7]`. H1 fixes the measurement; do it first.
2. **Real barge-in coverage is ~zero.** `barge_in_latency` n=3, all ~0.0002s, from 2 runs; **0/47** LLM calls cancelled. Barge-in **never once cancelled real in-flight work** in 20 runs — all 293 `cancel_all` calls logged "0 active, 0 queued, 0 pending". The only thing detection did in the field was self-interrupt on TTS echo (134 detections in run-220149). The cut machinery is correct but **unvalidated by a genuine user interrupt**.
3. **The headline "prior hardening" baseline cannot have come from this branch.** `tools/stress.py:282` (`scn_real`, the only real-model scenario) passes `warm_on_start=True` to a constructor that doesn't accept it (verified) → **`TypeError` before it measures anything**. The "warm p50 182ms" + 1000-turn/300-bargein/5000-flood PASS claims came from EchoLLM scenarios or a stale branch — treat them as **unreproducible on `stt-quality`**.
4. **Capacity counts liveness, not progress `[dist gap #7]`.** `at_capacity()` (`tasks.py:143-149`) uses `Thread.is_alive()`; `_reap_dead_locked` never joins (`:157`). A thread wedged forever in a capability call counts as healthy capacity — 6 such wedges report "at capacity, queue the rest" while doing zero work. The "never gets stuck" goal can fail silently with the cap reporting green. Not yet observed (peak concurrency was 3), but untested under a non-returning capability (the scenario that would test it is the broken `scn_real`).
5. **`OllamaLLM` is bounded (60s) but with no aggressive deadline.** `llm.py:142` default `timeout=60.0`, factory doesn't override (`llm_factory.py:80`). So the desktop fast tier self-heals at 60s — but that's **50s past the watchdog's 10s warning**, stranding a slot. The bare `LlamaCppLLM` (phone profile) has **no socket timeout** — genuinely unbounded. Cancel is per-token only (`capabilities.py:121`), so a pre-first-token wedge can't be barged out.
6. **The phone profile is where the conclusions invert, and it is never measured.** All 20 runs are 4090/Ollama/gemma3:4b. On `phone`: llama.cpp TTFT is 407ms→1330ms (vs desktop p50 0.06s), the LLM wedge is unbounded, and `phone_lite` puts the blocking input_gate/cleanup gates on the capture thread exactly where CPU is slowest. No replay/bench run on phone exists.
7. **Concurrent users via `remote/` are completely unexercised.** `remote/worker.py` = one brain / one `EventBus` / one `TaskRuntime` per room, sharing the global 6-task ceiling. Barge-in calls a **global** `supervisor.cancel_all()` with no session scoping — two participants would cancel each other's work. No multi-user test exists.
8. **STT accuracy has no number.** Transcripts are visibly garbled (`"Hello our listening to me"`, `"A bosom boom"`) — the *reason this branch exists* — yet no WER/CER is recorded in any `summary.json`. Garbled prompts degrade the whole pipeline regardless of latency.

## 8. Appendix: refuted / already-mitigated claims (checked and dismissed)

- **"Streaming TTS is dormant in production" `[lat-3]`** — **Refuted.** `config.json:235` `tts.streaming=true` is top-level, survives deep-merge into *every* profile; both production entrypoints (`app.py:422`, `remote/worker.py`) enable it. Only tests pass `stream_tts=False` (deliberately, to exercise both paths). Residual: flipping the *code* default to True is defensive cleanup; `drain_complete_sentences` lacks an abbreviation guard (quality nit, not latency).
- **"Router can't escalate at all / main-tier untested" `[lat-5, dec-2]`** — **Partly refuted, with a correction.** On the *uncommitted working tree*, `"Send me a long story"`/`"write a poem"` route to MAIN (the new `_GENERATION_MARKERS` + threshold=0.3), and 92 routing tests pass. **Correction:** those markers are **working-tree only** — `git show HEAD:core/routing.py` has 0 hits — so the committed code and all 20 recorded runs are genuinely fast-only (this is why B1 remains a blocker for the *shipped* artifact). The survivor: no recorded session yet captures a real 12b turn.
- **"No AEC ⇒ barge-in only fires on echo; half-duplex has no path to interrupt" `[int-1]`** — **Severity reduced (med, not blocker).** Half-duplex + no-AEC is real (`sherpa.py:632-651`). But the current branch *does* have an identity discriminator (`speaker_gate_input=true`, `barge_in_enabled=true`); the cited echo storms ran with input-gating off (older config), and the working interrupt path is test-passing. AEC is already a tracked P2 roadmap item, not an undiscovered blocker.
- **"Identity gate needs a margin OR-fallback" `[int-2]`** — **Refuted; the proposed fix would regress.** The absence of the OR is the deliberate `8d987b1` decision; the loud TTS echo *does* clear the 6dB margin (regression-tested), so an OR would reintroduce the 134× self-interrupt storm. All 26 barge-in tests pass.
- **"`intent_kind` can never escalate semantically" `[dec-2]`** — **Refuted.** `HeuristicRouter.score()` escalates assistant-mode queries via three text-reading levers (complexity/generation/length) independent of `intent_kind`; tested and passing. Genuine residue is narrow (single-marker short reasoning questions score 0.18 → fast, arguably correct for a voice assistant).
- **"Local LLM clients have no timeout; 6 hangs strand all slots" `[dist-1]`** — **Severity reduced (low).** `OllamaLLM` defaults to a finite `timeout=60.0` (`llm.py:142`) and self-heals; desktop sets `num_predict=768`. Only the bare `LlamaCppLLM` (non-default phone profile) is truly unbounded. Zero slot-stranding observed (peak concurrency 3/6). Real residual: 60s ≫ the 10s watchdog deadline (folded into roadmap #8).
- **"Queue-flood orphans `on_done`; EventBus single-consumer HOL-blocks the stop path" `[dist-4, dist-5]`** — **Confirmed in code, immaterial in production.** The orphan is real under the stress harness (`sherpa.py:758` drops oldest without firing `on_done`) but the only production caller (`runtime.py:332`) passes **no `on_done`** — nothing to orphan. The bus stop path is **off-bus** (`runtime.py:254-256`), TTS playback is a non-blocking `put_nowait` into a bounded(64) queue with synth on a separate thread, and `CONTROL_STOP` is priority 0. Both are low-severity defense-in-depth (bound the queue, document the fast lane), not current defects.
