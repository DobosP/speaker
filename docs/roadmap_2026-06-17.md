Actionable sequencing of the findings in
[`architecture_audit_2026-06-16.md`](architecture_audit_2026-06-16.md), under two
**owner decisions** that override the audit's default priorities:

- **D-A (barge-in is a MUST).** Open-speaker barge-in on the bare laptop speaker is
  a hard requirement (CLAUDE.md). It is promoted from "strategic" to a **protected
  track that runs through every phase**, and the *first* thing we do is measure
  where it actually stands (the audit found it field-unvalidated: zero real
  LLM-cancellations recorded).
- **D-B (keep dev artifacts).** The `logs/runs/` `*.wav` / `*.summary.json` /
  `*.txt` bundles stay committed **for now** — they're in active development use.
  The run-bundle PII purge + transcript-redaction is therefore **deferred to a
  pre-release gate** (last section), *not* a current action. These same WAVs become
  the barge-in golden-regression corpus in Phase 0, so keeping them is aligned with
  the plan.

Each phase lists **items (audit IDs)**, the **security guardrail** that must land
with it, and a **measurable exit criterion**. Phases are ordered by leverage and
dependency; you can also execute by track (index at the end).

## Phase 0 — Measure the baseline & build the safety net
*Change nothing yet; you can't protect a must-have you haven't measured.*

- **Enroll speaker-ID** (the ~2-min owner action). Unblocks the personal-VAD barge
  veto (Phase 2) and addressing; also makes the owner-gated action chokepoint
  operative.
- **Pin barge-in golden regressions** from the existing replay WAVs — especially
  `run-20260605-204551` (the documented self-interrupt + rejected-real-barge
  bundle). `python -m core --engine replay --replay-dir logs/runs` is already the
  mechanism. *(This is why D-B's "keep the WAVs" is correct.)*
- **Live open-speaker barge-in A/B on the bare Realtek speaker**: record real
  talk-over, measure **self-interrupt rate** (assistant cuts itself on its own TTS)
  and **missed-barge rate** + **cut-off latency** on a real talk-over.
- **Real low-spec latency run**: `python -m tools.bench --profile phone` and
  `--profile cpu_laptop` on real/throttled hardware → replace the specsim *estimates*
  with measurements.
- **Exit:** a barge-in scorecard (self-interrupt % / missed-barge % / cut latency)
  + a low-spec latency scorecard + a green replay-regression gate. Every later
  barge-in change is judged against this scorecard.

## Phase 1 — Adaptation spine + protect the barge-in real-time path
*Low risk, high leverage. Two of these exist to keep the capture/AEC path alive on
weak hardware — i.e. they are barge-in work.*

- **device-adapt-1** — auto-apply the probed profile at launch; **fail-fast** on an
  unknown `--device` (today it silently no-ops to the heavy `desktop` tier).
- **cross-platform-8** — validate `--device` names in `core.app` + `remote/worker.py`.
- **llm-inference-5** — make the phone llama.cpp path *runnable*: add
  `llama-cpp-python` extra, fetch GGUFs in `setup_models --gguf`, fail-fast with the
  exact command. (Foundational — the low-spec LLM path is inert without it.)
- **llm-inference-2** — wire `llm.n_threads` from the profile and **budget it against
  sherpa's capture/STT/TTS threads** so the LLM can't starve the real-time audio
  path. *(barge-in protection)*
- **audio-bargein-8** — move coherence/far-ring/level-EWMA ingest off the PortAudio
  callback (lock contention → real-time underrun, worst on small low-spec buffers).
  *(barge-in protection)*
- **Guardrail (mandatory, lands here):** the **cloud-enable invariant test** (no
  `device_profile` sets `cloud.enabled=true` or relaxes the actuator/speaker-ID
  gate) + the **`config.local.json` write allowlist** (persist profile name + model
  paths only). These make auto-profile selection provably safe.
- **Exit:** an unconfigured phone/cpu_laptop boots into the correct small-model tier
  automatically; capture-path starvation gone; barge-in scorecard **not regressed**;
  invariant test green in CI.

## Phase 2 — Barge-in robustness (the MUST)
*Each item lands behind the Phase-0 golden regressions **and** is validated on the
bare Realtek speaker across desktop + ≥1 low-spec device before merge.*

- **audio-bargein-2** — wire the **already-existing** runtime AEC-delay
  auto-calibration (`EchoCoherenceDetector.measured_delay_samples`) into a slow,
  debounced re-align. Directly attacks the self-interrupt root cause (static
  per-machine `aec_ref_delay_ms` vs the real ~260 ms delay) and removes the manual
  `echo_probe` step on every fresh/low-spec install.
- **audio-bargein-1** — per-device-profile audio-tuning block (`block_sec`,
  coherence `nperseg`/`ring_ms`, AEC taps, DTD K). Enabler: a phone currently runs
  *identical* barge DSP params to a 4090.
- **asr-tts-4** — decouple loudness normalization from streaming synth so first-audio
  streaming works **with** the echo floor (chunk-wise limiter, not whole-clip RMS).
  Cuts first-audio latency *and* keeps a stable echo level for the barge detector.
- **audio-bargein-5** — promote the implemented **Smart Turn v3** prosody endpointer
  (8 MB int8) from OFF to a per-profile default; needs the small **real-voice
  validation harness** (it's flat on synthetic TTS). Lets `endpoint_min_silence`
  drop, reclaiming part of the ~806 ms endpoint p50.
- **audio-bargein-6** — personal-VAD (pVAD) **veto** so a TV / bystander / leaked TTS
  can't trigger barge-in; reuses the enrolled speaker embedding from Phase 0,
  default-deny.
- **Guardrail:** keep the deterministic coherence detector as the primary barge
  trigger; treat the real-voice harness output as the merge gate.
- **Exit:** measured self-interrupt rate **down** and real-barge cut-off latency
  within target on the **open speaker**, on desktop + a low-spec device, with the
  golden regressions green. This is the acceptance bar for D-A.

## Phase 3 — On-device LLM speed + ASR latency (low-spec perf)
- **llm-inference-1** — reuse llama.cpp KV/prefix cache for the byte-stable system
  prompt (retain the pre-warm KV instead of re-prefilling each turn). **Biggest TTFT
  cut on the CPU phone path.**
- **asr-tts-1** — per-tier ASR policy (`phone_lite` → streaming-only finals +
  `greedy_search`; keep beam+SenseVoice on desktop).
- **asr-tts-2** — run the ASR second pass **asynchronously** (dispatch streaming
  final immediately, upgrade in place).
- **llm-inference-3** — cap on-device `max_tokens` (llama.cpp key, not `num_predict`).
- **llm-inference-9** — KV-cache quantization (INT8) + bounded context; keep
  speculative/NPU decode **gated to the high tier only**.
- **routing-cascade-4** — fully-local embedding/semantic router as the cheap
  disambiguator on weak profiles (capability router is OFF on every non-desktop
  profile today).
- **routing-cascade-1** — perplexity/confidence escalation (free signal; most turns
  resolve locally).
- **routing-cascade-7** — word-boundary-compile the marker scanners.
- **control-plane-2** — load-elastic admission (drop the task ceiling to 1 under
  load). **control-plane-3** — EWMA-scaled watchdog deadlines. **device-adapt-5** —
  tier-aware headroom signal (CPU% on llamacpp tiers).
- **Guardrail:** any cloud escalation (`routing-cascade-1/2`) keeps the regex+Luhn
  egress redactor as the unconditional last stage; speculation stays local-only.
- **Exit:** measured TTFT/turn latency on phone/cpu_laptop improved vs the Phase-0
  scorecard; the S-effort quick-win cluster shipped.

## Phase 4 — Dynamic, in-session adaptation & resilience
*Closes the "choose-once-at-launch" gap — the field's table-stakes the audit said
you're a generation behind on.*

- **device-adapt-2** — thermal + battery sensing; size for the **sustained (hot)
  plateau**, not the cold burst (critical for fanless phones).
- **asr-tts-3** — runtime RTF guard: self-degrade when decode/synth falls behind
  real-time.
- **routing-cascade-2** — activate the dormant, fail-safe `live_routing` nudge on
  slow profiles. **device-adapt-9** — calibrate specsim/recommend_profile thresholds
  from the Phase-0 real runs.
- **llm-inference-6** — reap timeout / cancellation for in-process llama.cpp
  (swap/OOM stalls). **routing-cascade-6** — compute the route decision once per turn
  (aq-5).
- **device-adapt-7** — lift the 2..4 sherpa thread clamp (bench-gated).
  **llm-inference-7** — per-device GGUF K-quant ladder (Q3 phone_lite / Q4 phone).
- **Guardrail:** thermal/battery readings are bucketed (nominal/warm/hot) in run
  bundles — no raw curves; live-routing lever flips stay owner-consented, not
  shipped-on.
- **Exit:** the device measurably self-tunes within a session under thermal/load
  pressure; no unbounded inference stalls.

## Phase 5 — Cross-platform reuse + deep barge-in tier
- **audio-bargein-9** → **cross-platform-4** — extract the barge-in cascade into a
  pure `BargeInDetector` policy class (aq-1/D7) and reuse it in the LiveKit engine,
  so the lowest-spec targets inherit the tuned P1 logic instead of re-deriving it.
- **cross-platform-2** — mobile barge-in **self-calibration** from a short
  ambient/echo sample (replace the hardcoded `_bargeInRms=0.08`). *(barge-in, phone
  form factor)* **cross-platform-1** — port device-profile tiering into the Flutter
  app.
- **audio-bargein-7** — land the deferred **DTLN-aec** deep tier at the phone-sized
  128-unit config. **audio-bargein-4** — TEN VAD as a faster-transition option.
  **audio-bargein-3** — iAEC spike (research-grade; only if Phase 2 leaves residual
  self-interrupts). **audio-bargein-10** — anticipatory endpointing to mask slow-CPU
  latency.
- **cross-platform-5/6/7** (Dart golden contract, anti-aliased LiveKit resampling,
  LAN/self-host-default privacy), **control-plane-7** (structured cancel-scope),
  **security-privacy-2/4** (tiered NER backstop OFF on phone; attestable cloud tier).
- **Exit:** one validated barge-in policy shared across desktop / LiveKit / mobile;
  deep AEC tier shipped; remote path emits metrics + scoped privacy.
